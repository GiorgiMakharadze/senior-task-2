# Code Review: Broken Subscription Implementation

## Issue 1: Сервис напрямую зависит от *sql.DB
- Категория: Нарушение слоистой архитектуры
- Severity: CRITICAL
- Проблема: `SubscriptionService` хранит конкретный `*sql.DB`, жёстко связывая бизнес-логику с SQL-инфраструктурой. Нет ни интерфейса репозитория, ни абстракции, отделяющей core от infra.
- Почему это важно: Usecase-слой невозможно полноценно тестировать без реальной базы данных. Замена хранилища или добавление кэширования потребует переписывать бизнес-логику. Это нарушает dependency inversion — core должен зависеть от абстракций, а не от инфраструктуры.

## Issue 2: Сервис напрямую зависит от *http.Client
- Категория: Нарушение слоистой архитектуры
- Severity: CRITICAL
- Проблема: `SubscriptionService` хранит конкретный `*http.Client` и напрямую делает HTTP-вызовы для валидации клиента и оформления refund.
- Почему это важно: Бизнес-логика связана с деталями HTTP-транспорта (сборка URL, декодирование JSON, обработка response). Такой код невозможно нормально тестировать без живого HTTP-сервера или test server. Внешний billing API должен быть скрыт за интерфейсом.

## Issue 3: HTTP-вызов для валидации клиента встроен в бизнес-логику
- Категория: Нарушение слоистой архитектуры
- Severity: CRITICAL
- Проблема: `CreateSubscription` напрямую вызывает `s.client.Get(...)` для валидации клиента. URL захардкожен, а HTTP-вызов смешан с доменной логикой.
- Почему это важно: Домен и usecase-логика не должны знать о HTTP. Здесь в одном методе перемешаны транспорт, сериализация и бизнес-оркестрация. Код становится плохо тестируемым и жёстко привязанным к конкретной форме внешнего API.

## Issue 4: HTTP-вызов для refund встроен в бизнес-логику
- Категория: Нарушение слоистой архитектуры
- Severity: CRITICAL
- Проблема: `CancelSubscription` напрямую вызывает `s.client.Post(...)`, собирает JSON и делает внешний API-вызов inline.
- Почему это важно: Побочный эффект refund спрятан внутри логики отмены, при этом нет нормальной обработки ошибок, retry-стратегии и разделения между state transition и внешним вызовом. Если refund-вызов тихо упадёт, подписка всё равно останется в статусе cancelled без какого-либо следа о неуспешном возврате.

## Issue 5: Нет доменного агрегата — Subscription представляет собой анемичный data bag
- Категория: Чистота домена
- Severity: CRITICAL
- Проблема: `Subscription` — это обычная структура со всеми публичными полями и без методов. Нет проверки инвариантов, контролируемых переходов состояния и инкапсуляции.
- Почему это важно: Любой код может записать в любое поле любое значение, обходя бизнес-правила. Нет гарантии, что подписка проходит только через допустимые состояния. Такой “domain” по сути является DTO без поведенческой ответственности.

## Issue 6: Нет доменных событий
- Категория: Чистота домена
- Severity: WARNING
- Проблема: Создание и отмена подписки не порождают domain events. Нет ни отслеживания изменений, ни возможности event sourcing.
- Почему это важно: Downstream-процессы (уведомления, аналитика, обработка refund, аудит) не могут реагировать на изменения жизненного цикла. Система не готова к event-driven architecture или outbox-паттерну.

## Issue 7: Status — это raw string без type safety
- Категория: Чистота домена
- Severity: WARNING
- Проблема: `Status` объявлен как `string`, а значения вроде `"ACTIVE"` и `"CANCELLED"` разбросаны по коду строковыми литералами.
- Почему это важно: Любая опечатка становится тихим багом. Компилятор никак не гарантирует корректность допустимых статусов. Добавление новых состояний превращается в поиск строк по всему проекту.

## Issue 8: Генерация UUID и вызов time.Now() внутри бизнес-логики
- Категория: Чистота домена
- Severity: WARNING
- Проблема: `CreateSubscription` напрямую вызывает `uuid.New().String()` и `time.Now()`. Это скрытые инфраструктурные зависимости внутри бизнес-слоя.
- Почему это важно: ID и timestamp становятся недетерминированными, из-за чего тесты нельзя воспроизводить стабильно. Бизнес-слой начинает владеть тем, чем владеть не должен. Генерация ID и источник времени должны внедряться через зависимости.

## Issue 9: Игнорируется ошибка HTTP GET при валидации клиента
- Категория: Обработка ошибок
- Severity: CRITICAL
- Проблема: `resp, _ := s.client.Get(...)` отбрасывает ошибку. Если HTTP-вызов не удался, `resp` будет nil, и вызов `resp.Body.Close()` на следующей строке приведёт к nil pointer panic.
- Почему это важно: Помимо самого падения, игнорирование ошибки означает, что network failure, DNS issue или timeout неявно превращаются в “customer invalid” (так как `result.Valid` по умолчанию будет `false`). В результате валидные клиенты отклоняются из-за инфраструктурных проблем — это уже неверное бизнес-решение из-за схлопнутой семантики ошибок.

## Issue 10: Игнорируются ошибки row.Scan, json.Decode, json.Marshal и HTTP POST
- Категория: Обработка ошибок
- Severity: CRITICAL
- Проблема: Несколько ошибок полностью игнорируются: `row.Scan(...)`, `json.NewDecoder(...).Decode(...)`, `json.Marshal(...)` и refund-вызов `s.client.Post(...)`. Ничего не проверяется.
- Почему это важно: Ошибка `Scan` означает, что код начнёт работать с нулевыми значениями (пустой ID, нулевая цена, epoch time в start date), что приведёт к неправильному расчёту refund и фантомным данным. Ошибка `POST` при refund означает, что деньги не вернутся клиенту и при этом никакого сигнала об ошибке не будет.

## Issue 11: Для денег используется float64
- Категория: Работа с деньгами
- Severity: CRITICAL
- Проблема: `Price` имеет тип `float64`, а refund считается через floating-point арифметику: `sub.Price * (30 - daysUsed) / 30`.
- Почему это важно: Floating-point не может точно представлять десятичные денежные значения. Выражения вроде `29.99 * 20 / 30` дают ошибки округления, которые накапливаются на большом числе транзакций. Финансовые расчёты должны использовать integer cents (`int64`) или decimal type, чтобы избежать тихих потерь или лишнего начисления денег.

## Issue 12: Для расчёта бизнес-даты используется time.Since()
- Категория: Тестируемость
- Severity: CRITICAL
- Проблема: `CancelSubscription` использует `time.Since(sub.StartDate)`, а это внутри обращается к wall clock. Результат меняется при каждом запуске.
- Почему это важно: Расчёт refund становится недетерминированным и нетестируемым. Нельзя написать надёжный тест с точной проверкой суммы возврата, потому что прошедшее время меняется между запусками. Бизнес-вычисления по времени должны работать через внедрённый clock.

## Issue 13: Нет интерфейсов — тестирование без реальной инфраструктуры невозможно
- Категория: Тестируемость
- Severity: CRITICAL
- Проблема: `SubscriptionService` принимает конкретные `*sql.DB` и `*http.Client`. Нет интерфейсов для репозитория, billing client или clock.
- Почему это важно: Для unit-тестов потребуется реальная база и HTTP-сервер. Такие тесты становятся медленными, flaky и зависящими от окружения. Нормальная тестируемость достигается через dependency inversion и интерфейсы, которые можно подменять фейками.

## Issue 14: SELECT * — хрупкое решение
- Категория: Нарушение слоистой архитектуры
- Severity: WARNING
- Проблема: `CancelSubscription` использует `SELECT * FROM subscriptions` и полагается на то, что позиционный `Scan` совпадёт с порядком колонок.
- Почему это важно: Добавление, удаление или перестановка колонок в таблице тихо ломает `Scan`. В запросе нужно явно перечислять колонки, чтобы код был жёстко согласован со схемой.

## Issue 15: Refund может стать отрицательным
- Категория: Работа с деньгами
- Severity: WARNING
- Проблема: Формула `sub.Price * (30 - daysUsed) / 30` даёт отрицательное значение, если `daysUsed > 30`. Нет ни clamping, ни проверки границ.
- Почему это важно: Отрицательный refund может быть интерпретирован как дополнительное списание денег с клиента. Расчёт должен жёстко ограничиваться нулём, если billing period уже полностью истёк.

## Issue 16: CustomerID напрямую вставляется в URL без escaping
- Категория: Обработка ошибок
- Severity: WARNING
- Проблема: `"https://api.billing.com/validate/" + req.CustomerID` конкатенирует пользовательский ввод прямо в URL path без escaping и валидации.
- Почему это важно: Если `CustomerID` содержит slash, query parameters или специальные символы, URL может сломаться, уйти не в тот endpoint или открыть путь для injection. Все пользовательские данные, попадающие в URL, должны корректно экранироваться.
